# AUTONOMY CHARTER（テンプレート）

> AIに「何を任せ、何を任せないか」を明文化する、権限の憲法。
> 完璧である必要はありません。境界が書かれていることが、検出と改善の前提条件です。
> コピーして、あなたの環境に合わせて書き換えてください。所要30分。

---

## 0. 適用範囲

- **対象AI**:
  - 設計レーン: （例: 対話型AI — 要件定義・判断・レビューに使用）
  - 実行レーン: （例: エージェント型AI — 実装・ファイル操作・git操作に使用）
- **対象環境**: （例: 個人開発環境のみ。業務環境・本番データは対象外）
- **改訂権限**: 人間のみ。AIは本Charterの変更を**提案**できるが、**実行**できない。

## 1. Green ── 自律実行してよいこと

承認なしで実行できる操作。迷ったらYellowに置く。

- [ ] 読み取り専用の調査・分析・要約
- [ ] 作業ディレクトリ内のファイル作成・編集
- [ ] テストの実行、lint、フォーマット
- [ ] 作業ブランチへのコミット
- [ ] （あなたの環境に合わせて追加）

## 2. Yellow ── 人間ゲートが必要なこと

実行前に、人間の明示的な承認を要する操作。
**承認は「計画の提示 → 人間の注釈 → 承認」の3ステップで行う。**

- [ ] mainブランチへのマージ
- [ ] 外部公開に相当する操作（公開リポジトリへのpush、デプロイ、SNS投稿）
- [ ] 外部サービスへの書き込み（メール送信、API経由のデータ更新、チケット起票）
- [ ] 費用が発生する操作（しきい値: ¥＿＿＿ 以上）
- [ ] 依存パッケージの追加・更新
- [ ] （あなたの環境に合わせて追加）

## 3. Red ── 禁止事項

いかなる承認があっても実行しない操作。

- [ ] 認証情報・APIキー・個人情報の出力、コミット、ログへの記録
- [ ] 情報Tier（別紙）に該当する情報の、公開領域への書き込み
- [ ] 本Charterおよびゲート機構そのものの変更・無効化・迂回
- [ ] 評価・テストコードの削除や無効化による「見かけ上の成功」の作出
- [ ] （あなたの環境に合わせて追加）

## 4. 違反時の手順

1. **停止** — 当該作業を中断する
2. **記録** — SCR（Self-Correction Record）を起票する → `templates/SCR.md`
3. **判断** — 原因がルールの曖昧さであれば、Charter改訂を人間が判断する

違反は失敗ではなく、境界の解像度を上げる材料です。
記録されない違反だけが、本当の失敗です。

## 5. レビューサイクル

- **定期**: 月次でGreen / Yellowの境界を見直す（Yellowで一度も止めなかった項目はGreen昇格を検討）
- **随時**: SCR起票のたびに、該当条項の妥当性を確認する

## 6. 情報Tier（別紙の要約）

| Tier | 内容 | 扱い |
|---|---|---|
| Tier 1 | 個人特定情報（実名、連絡先、ID類） | 公開領域への書き込み禁止 |
| Tier 2 | 所属・業界が特定される情報 | 公開時は一般化必須 |
| Tier 3 | 内部名称・関係者名・取引先名・未公開案件 | 公開領域への書き込み禁止。CIでパターン検査 |
