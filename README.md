# AI Autonomy Playbook

**自動化の前に、統治を。**

AIツールを「使える」ことと、組織がAIで「自走できる」ことは、別の能力です。
前者は個人のスキルで、後者は運用の設計です。
そして企業のAI活用が止まる場所は、ほぼ常に後者です。

このリポジトリは、私が自分自身を最初の実験体として構築・運用している
**個人規模のAI統治システム**の公開版です。
一人の事業側PMが、対話型AIとエージェント型AIを「組織」として運用し、
企画から実装・リリースまでを回すための権限設計・品質管理・自己修正の仕組みを、
テンプレートと運用実例つきで提供します。

> エンジニアリング組織のためのものではありません。
> **コードを書かない側から、AI活用を組織能力として立ち上げる人**のためのものです。

---

## なぜこれが必要か

AI活用の現場で繰り返し起きることは3つです。

1. **属人化** — 「あの人はAIをうまく使う」で止まり、組織の能力にならない
2. **デモ止まり** — PoCは動くが、権限・品質・例外処理が未設計でオペレーションに載らない
3. **統治不在** — 何をAIに任せてよいか誰も決めておらず、リスク部門の一言で全停止する

モデルの性能はもう本質的な制約ではありません。制約は、
**AIに何を任せ、何を任せず、間違えたときにどう直すかの運用設計**が存在しないことです。
このプレイブックは、その運用設計を最小単位（一人）から始められる形にしたものです。
一人で回るものは、チームに拡張できます。逆は成立しません。

## 全体像

```
┌──────────────────────────────────────────────┐
│  AUTONOMY CHARTER ── 権限の憲法               │
│  Green: AIが自律実行できること                │
│  Yellow: 人間ゲートが必要なこと               │
│  Red: 禁止事項                                │
└────────────────────┬─────────────────────────┘
                     │
        ┌────────────┴────────────┐
        │        役割分離          │
        │  設計レーン: 対話型AI    │ ← 要件定義・判断・レビュー
        │  実行レーン: エージェント│ ← 実装・ファイル操作・git
        └────────────┬────────────┘
                     │
┌────────────────────┴─────────────────────────┐
│  タスクボード ── 5レーン運用 / WIP制限        │
└────────────────────┬─────────────────────────┘
                     │
        ┌────────────┴────────────┐
        │  Gate設計 ── 着手判断    │ Gate 0: go / no-go
        └────────────┬────────────┘
                     │
┌────────────────────┴─────────────────────────┐
│  SCR: Self-Correction Record                 │
│  失敗 → パターン化 → ルール更新               │──→ Charterへ還流
└──────────────────────────────────────────────┘
```

統治（Charter）が頂点にあり、失敗の記録（SCR）が統治へ還流する。
この閉ループが「自走」の正体です。

## コンポーネント

### 1. AUTONOMY CHARTER ── 権限の憲法

AIに「何を任せ、何を任せないか」を、Green / Yellow / Red の3層で明文化します。
重要なのは網羅性ではなく、**境界が書かれていること**です。
境界が文書化されていれば、違反は検出でき、検出できれば改善できます。

→ [`templates/AUTONOMY-CHARTER.md`](templates/AUTONOMY-CHARTER.md)

### 2. 役割分離 ── 設計と実行を混ぜない

- **設計レーン**（対話型AI）: 要件定義、選択肢の比較、判断の壁打ち、成果物レビュー
- **実行レーン**（エージェント型AI）: 実装、ファイル操作、テスト実行、git操作

同じ画面で混ぜると、判断の根拠と実行の痕跡が混線し、監査できなくなります。
分けると、「なぜそうしたか」（設計レーンのログ）と「何をしたか」（実行レーンのコミット）が
独立に追跡できます。これは人間の組織で企画と開発を分けるのと同じ理屈です。

### 3. タスクボード ── 5レーン運用

Backlog → Ready → In-Progress → Review → Done。
エージェントはReadyからタスクを「請け負い」、完了時に必ずReviewを経由します。
WIP制限により、エージェントの並列暴走（やりかけタスクの山）を構造的に防ぎます。

### 4. Gate設計 ── 「作らない」を決める仕組み

着手前に Gate 0（価値仮説・データ入手性・リスクの3点審査）で go / no-go を判定します。

運用実績: 直近の候補3案件のうち、**GOは1件。1件はGate 0でクローズ、1件はスコープを限定して継続。**
統治とは、作る判断ではなく、**作らない判断が記録つきでできること**です。

### 5. SCR ── 失敗を資産にする最小単位

AIの失敗・誤判断を「事実 → 原因（直接/構造）→ 修正 → ルールへの還流先」の形式で記録します。
1件のSCRは反省文ですが、2件目から組織学習が始まります。

→ [`templates/SCR.md`](templates/SCR.md)

**実例**: RAGシステムの評価ハーネスで、ホールドアウト除外ロジックが
無効化（no-op）したまま「評価が通り続けていた」バグを、結果が良すぎることへの違和感から手動検証で検出。
「評価が通っている」ことと「評価が機能している」ことは別である、という教訓をSCR化し、
**評価系コードの変更には、評価が壊れていることを検出するテストを伴う**ことをCharterに還流しました。

### 6. 情報Tier ── 公開してよい情報の階層

- **Tier 1**: 個人特定情報
- **Tier 2**: 所属・業界が特定される情報
- **Tier 3**: 内部名称・関係者名・取引先名

公開リポジトリはCIでパターン検査を実行し、Tier該当情報の混入を機械的に検出します。
ガバナンスを「気をつける」で済ませず、検査可能な仕組みに落とすことが、
企業環境でAI活用を止められないための最低条件です。

なお、検査パターンの平文をリポジトリに置くと、**パターン自体がTier情報になる**という自己矛盾が生じます。
本リポジトリの検査はソルト付きハッシュ照合（HMAC-SHA256、ソルトはCI Secretに格納）で行い、この欠陥を回避しています。
→ [`scripts/tier_check.py`](scripts/tier_check.py) / [`scripts/make_patterns.py`](scripts/make_patterns.py)

## 運用実績（匿名化済み）

本体制（設計レーン＋実行レーン＋Charter統治）下での実績です。
所属組織・取引先の固有名詞は含みません。数値は桁感を保ちつつ一般化しています。

- 複数の業務システム（データウェアハウス／BIダッシュボード／ノーコードアプリ／
  スプレッドシート／SFA／オンプレミス由来データ）を単一のデータ基盤に統合し、
  全社の意思決定インフラとして構築・運用。企画・データ設計・実装・運用までを単一担当で担う
  → 部門・ツールごとに分断していたデータを一本化し、「作る人／使う人」の壁を解消した

- 利用者は経営企画・営業統括・オペレーション・マーケティング・法務・情報セキュリティなど
  全社の主要部門にわたり、各部門が集計値と対象アカウントを確認して日次業務の判断に用いる。
  営業現場のコアユーザーは外部委託を含め約120名規模
  → 個別の集計依頼が常態化していた状態を、部門横断のセルフサービス型基盤に置き換えた

- B2Bアカウント10万規模の、契約・決済・取引先マスタ・取引実績にまたがる多面的データを統合。
  契約者別／グループ別／個別の異なる粒度での集計を、単一パイプラインで供給

- 外部BPOパートナーを含むオペレーション全体を対象に、訪問対象の抽出から
  現場での活用（訪問計画・情報配信）までを設計
  → データ分析の成果を、現場の行動に接続するところまで一気通貫で実装した

- 業務データパイプラインを、設定／処理／通知／運用／テストを分離した構成で継続運用
  → 障害時の切り分けを局所化し、単一担当でも壊れにくく運用が回る状態を実現した

- サーバーレス構成のOSSプロダクトを設計・実装・公開

- RAGシステムの評価ハーネスで、ホールドアウト除外が無効化したまま「評価が通り続けていた」
  バグを、結果への違和感から検出・修正
  → 「評価が通っている」ことと「機能している」ことは別である、という教訓をSCR化した

## このリポジトリ自体について

本リポジトリは、ここに書かれた統治システム自身の管理下で作成されています。
人間（設計・判断・注釈）とAI（起草・実装）の役割分離を守り、
計画 → 人間による注釈 → 実行のループを回し、その過程を [`decision-log/`](decision-log/) に収録しています。

**方法の主張ではなく、方法の実演として。**

## 使い方（最短ルート）

1. [`templates/AUTONOMY-CHARTER.md`](templates/AUTONOMY-CHARTER.md) をコピーし、あなたのAIの権限3層を埋める（30分で十分です）
2. 設計レーンと実行レーンを物理的に分ける（ツールを分ける。同じ画面で混ぜない）
3. 最初の失敗が起きたら [`templates/SCR.md`](templates/SCR.md) で記録する

完璧な統治を最初に作る必要はありません。**境界を1本引き、破られたら記録し、記録から直す。**
それだけで、AI活用は属人スキルから運用システムに変わり始めます。
