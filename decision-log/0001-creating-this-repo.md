## 1. 計画（設計レーンで作成）

### 1-1. リポジトリ構成

```
ai-autonomy-playbook/
├── README.md                      # 本体
├── LICENSE                        # MIT（文章はCC BY 4.0、READMEに記載）
├── .gitignore                     # 平文パターンファイルを除外
├── .tier-patterns.sha256          # Tierパターンのハッシュ表（平文は置かない）
├── templates/
│   ├── AUTONOMY-CHARTER.md        # 権限憲法テンプレート

├── decision-log/

├── scripts/
│   ├── make_patterns.py           # 平文リスト→ハッシュ表の生成
│   └── tier_check.py              # ハッシュ照合スキャナ
└── .github/workflows/
    └── tier-check.yml             # push/PR時の自動検査
```

### 1-2. 主要な設計判断

1. **Tier検査をハッシュ照合方式（HMAC-SHA256＋Secret管理のソルト）で実装する**
   - 根拠: 検査パターンの平文をリポジトリに直書きすると、パターン自体がTier情報になる
     という自己矛盾（既知の設計欠陥）が生じるため。ハッシュ表のみを公開し、
     ソルトをローカル環境変数とGitHub Actions Secretに分離保管することで解消する。
   - 既知のトレードオフ: パターンの文字数は表から推測可能（許容）。
     フォークからのPRではSecretが渡らず検査が失敗する（個人リポのため許容）。
2. **CIログでは一致内容を表示しない**
   - 根拠: 公開リポジトリのActionsログは誰でも閲覧できるため、
     検出された機密語そのものをログに出すと検査が漏洩経路になる。
     表示はファイル名・位置・長さに限定し、内容確認はローカルのみ（`TIER_CHECK_REVEAL=1`）。
3. **公開pushはYellowゲートとし、人間の明示承認後にのみ実行する**
4. **初回コミットに本decision logを含める**
   - 根拠: 「最初のコミットから統治下にある」ことを履歴上の事実にするため。
     実行結果（コミットID・検査結果）は事後にしか確定しないため、2回目のコミットで追記する。

### 1-3. 実行手順（実行レーンへの指示の要約）

1. ローカルにリポジトリを配置し、`git init`
2. 人間が `tier-patterns.private.txt` を作成（git管理外）→ `make_patterns.py` でハッシュ表生成
3. `tier_check.py` をローカル実行 → **PASS（0件）を確認**（FAILなら公開中止・修正）
4. 初回コミット作成
5. **［Yellowゲート］人間の承認を待つ** → 承認後に `gh repo create --public` ＆ push
6. GitHub Actions Secret に `TIER_SALT` を登録、CI初回実行のPASSを確認
7. 本ファイルの「3. 実行ログ」を追記し、2回目のコミットをpush

- CI（tier-check）初回実行結果: **success（緑・6秒）** — Run: https://github.com/dkamehat/ai-autonomy-playbook/actions/runs/27422027770
  - 非ブロッキング注記: `actions/checkout@v4` / `actions/setup-python@v5` が Node.js 20 非推奨の警告（2026-06-16以降 Node24 強制予定）。動作・判定には影響なし。将来アクションのバージョン更新で解消する。
- 計画からの逸脱（理由とともに）:
  1. **公開手順の順序分離**: HANDOFF原文の `gh repo create --public --source=. --push`（作成と同時にpush）を、**①リポジトリ作成 → ②CI Secret(`TIER_SALT`)登録 → ③push → ④topics付与** の順に分離した。理由: push と同時にCIが起動すると、その時点で Secret 未登録のため `tier_check.py` が exit 2（ERROR）で失敗し、初回CIが赤になる。Secret を先行登録することで初回CIから緑を実現した。本順序分離は人間が 2026-06-12 に承認済み（§2 参照）。
  2. **ソルトの隔離運用**: `TIER_SALT` の値を実行レーン（Claude Code）のセッション・コマンド・ログに一切通さず、人間がソルト保持端末で `make_patterns.py` / `tier_check.py` / `gh secret set` を実行する運用とした。結果、実行レーンによるソルト値の表示・ファイル書き込みは発生していない（Red回避）。ただしソルトの世代管理は人間側で乖離が生じ、これは逸脱3で詳述する。
  3. **公開後に検査の実効性ギャップを検出・修正**: 公開直後、作業ツリーの `.tier-patterns.sha256` が CI Secret と別ソルトで再生成されていたことを起点に、「**緑のCIはソルト一致を保証しない**（Secretのソルトと公開ハッシュ表のソルトが不一致でも、ハッシュが永遠に一致せず PASS になる）」という no-op リスクを検出した。突き合わせの結果、公開済み表（HEAD・旧ソルト）と現行Secret（5代目ソルト）の**全37ハッシュが不一致**＝当初の緑CIは実効性を伴わない vacuous PASS だったと判明。修正として、現行Secretと同一ソルトで再生成した表を3回目コミットで公開し、**公開表 = Secret** に整合させた（CI緑を再確認）。本件は README記載の評価ハーネス no-op バグと同型であり、§4のSCRへ昇格する。

## 4. 結果と教訓

> （完了後に1〜3行で）

- 統治システムを当のリポジトリ自身に適用し、**計画 → 人間注釈(§2) → 実行 → 記録(§3)** のループを実走させて公開まで完遂した。
- 教訓: 検査をCI Secretに依存させる場合、**Secret登録を初回pushより前に置く**ことで「初回CIが赤 → 再実行で緑」という不要なノイズを構造的に回避できる（§3 逸脱1）。秘密情報（ソルト）は実行レーンに渡さず人間が保持・適用する分離が、Red違反の余地を最初から消す（§3 逸脱2）。
- SCR候補: 秘密値の受け渡し手順が画面転記・クリップボード競合を誘発した。修正＝表示工程の排除と、保存検証の手順化。→ `templates/SCR.md` で起票予定。
- SCR現実化: 上記候補が実障害として顕在化した — ソルト回転（〜5代目）と画面転記・クリップボード競合により、CI Secret と公開ハッシュ表のソルトが乖離し、検査が**緑のまま no-op 化**していた（§3 逸脱3）。「通るCI ≠ 機能するCI」。正式に `templates/SCR.md` へ起票する。
- 次回タスク予約（カナリア負テスト）: 「意図的に埋めた検出対象が CI で**必ず FAIL する**」ことを自動確認するテストを追加し、no-op（緑だが機能していない）を構造的に検出可能にする。本日はスコープ外、次タスクとして実施。
